# v2.1 Developer Power Tools â€” Design Spec

## Overview

Make clipass a programmable clipboard platform for power users. Add a CLI tool, pub/sub event system, tags, action chains, copy-append, and small QoL improvements. No competitor offers this combination â€” clipass becomes the only clipboard manager that's also developer infrastructure.

**Milestone:** v2.1 "Developer Power Tools"
**Scope:** 7 features across 3 layers (data, CLI, events) + 3 QoL improvements

## Architecture

Three layers build on each other:

1. **Data layer** â€” Tags (SwiftData model, many-to-many with ClipboardItem)
2. **CLI layer** â€” `clipass-cli` binary communicating with the app via Unix domain socket
3. **Event layer** â€” Pub/sub event streaming over the same socket

The Unix domain socket at `~/Library/Application Support/clipass/clipass.sock` serves dual purpose: CLI request/response and pub/sub event streaming, multiplexed by message `type`.

Small QoL features (copy-append, pause/resume, hide menu bar icon) are independent and can be built in any order.

## Feature 2: CLI Tool

### Binary & Installation

- **Build strategy:** CLI is a separate `executableTarget` in Package.swift named `clipass-cli`. It shares a `ClipassShared` library target with the app for socket protocol types and argument parsing. The app's build phase copies the CLI binary into `clipass.app/Contents/MacOS/clipass-cli`.
- Installation: `clipass setup-cli` symlinks to `/usr/local/bin/clipass` (or user runs `ln -s` manually)
- App offers setup on first launch or in Settings > General

### Transport

- Unix domain socket: `~/Library/Application Support/clipass/clipass.sock`
- App owns the socket, CLI is a thin client
- All data lives in the app's SwiftData store

### Item Addressing

CLI commands accept items by **1-based positional index** into recent history (most recent = 1). UUIDs are also accepted for scripting (e.g., from `clipass list --json` output or pub/sub events).

- `clipass get 3` â€” 3rd most recent item
- `clipass get a1b2c3d4-...` â€” by UUID
- `clipass list --json` output includes both `index` and `id` (UUID) fields

## Feature 3: Pub/Sub Event System

### Event Types

```json
{"event": "copy", "id": "uuid", "content": "...", "source_app": "com.apple.Terminal", "tags": [], "timestamp": "ISO8601"}
{"event": "delete", "id": "uuid", "timestamp": "ISO8601"}
{"event": "tag_added", "id": "uuid", "tag": "work", "timestamp": "ISO8601"}
{"event": "tag_removed", "id": "uuid", "tag": "work", "timestamp": "ISO8601"}
{"event": "transform", "id": "uuid", "rule": "trim-whitespace", "timestamp": "ISO8601"}
{"event": "pause", "timestamp": "ISO8601"}
{"event": "resume", "timestamp": "ISO8601"}
{"event": "clear", "count": 42, "timestamp": "ISO8601"}
```

**Content in events:** `copy` events include `content` by default. `transform` events omit before/after content by default (may contain sensitive data) â€” use `clipass listen --verbose` to include it. All events respect the app's existing redaction patterns.

### Socket Protocol

**Framing:** Length-prefixed binary framing: `<4-byte big-endian length><JSON payload>`. This avoids ambiguity with newlines in clipboard content. The JSON payload is always compact (no pretty-printing).

- Client connects, sends: `{"type": "subscribe", "filters": {...}}`
- Server streams: `{"event": ...}` messages
- CLI commands use: `{"type": "command", "cmd": "list", "args": {...}}` â†’ `{"type": "response", ...}`
- Both request/response and event streaming on same socket, distinguished by `type`

**`clipass listen` output:** The CLI converts framed messages to newline-delimited JSON on stdout for pipe-friendliness. Content fields with newlines are JSON-escaped (`\n` literals).

### Subscription Filters (via `clipass listen`)

- `clipass listen` â€” all events
- `clipass listen --event copy` â€” single event type
- `clipass listen --event copy,transform` â€” multiple types
- `clipass listen --tag work` â€” only events for tagged items
- `clipass listen --source-app Terminal` â€” filter by source app (substring match against both display name and bundle ID)

### Use Case Examples

```bash
# Auto-tag URLs
clipass listen --event copy | jq -r 'select(.content | test("^https?://")) | .id' | while read id; do
  clipass tag add "$id" url
done

# Log Slack copies
clipass listen --event copy --source-app Slack >> ~/slack-copies.log

# Desktop notification on copy
clipass listen --event copy | while read line; do
  osascript -e "display notification \"Copied\" with title \"clipass\""
done
```

## Feature 4: Action Chains

### Data Model

```
ActionChain (SwiftData @Model)
â”śâ”€â”€ id: UUID
â”śâ”€â”€ name: String (unique, kebab-case)
â”śâ”€â”€ displayName: String

```

### Built-in Step Types

| Type | Config | Example |
|------|--------|---------|
| `transform` | Existing transform rule name | Apply "strip-whitespace" |
| `trim` | `leading`, `trailing`, `both`, `blank-lines` | Remove blank lines |
| `case` | `upper`, `lower`, `title`, `camel`, `snake`, `kebab` | Convert to snake_case |
| `replace` | `pattern\|replacement` (regex) | `\t\|  ` (tabs to spaces) |
| `script` | Shell command (stdinâ†’stdout) | `sed 's/foo/bar/g'` |

### Usage

```bash
# CLI
pbpaste | clipass chain clean-code
clipass chain list
clipass chain run clean-code --input "messy text"
```

- **Overlay:** Right-click item â†’ "Apply Chain" â†’ select chain
- **Context menu:** Chains appear based on content type matching
- **Settings UI:** "Chains" tab â€” visual pipeline builder with add/reorder/remove steps and live preview

### Relationship to Existing Features

**Transforms:** Single regex rules that auto-apply on copy (reactive). Chains can include transforms as steps.
**ContextActions:** Shell commands triggered from the context menu with content filters. Chains are a superset â€” ContextActions remain as-is for simple one-off commands, chains are for multi-step pipelines. They coexist in the overlay context menu (chains under "Apply Chain", context actions under "Actions").

### Security

- `script` steps receive input via **stdin only** â€” no string interpolation of clipboard content into shell commands
- Chains are **manual-only** â€” they cannot be triggered automatically by pub/sub or copy events (prevents injection)
- Settings UI shows a warning icon on chains containing `script` steps

## Feature 5: Copy-Append

### Behavior

- Global hotkey (configurable, e.g., Cmd+Shift+C) via KeyboardShortcuts
- On trigger: set `appendPending` flag on ClipboardMonitor â†’ read pasteboard â†’ find most recent ClipboardItem â†’ append with `\n` separator â†’ update item in SwiftData â†’ clear flag
- **Race condition guard:** The `appendPending` flag tells the 500ms poll loop to skip creating a new item for the next pasteboard change. The flag auto-clears after 1 second as a safety net.
- No new history entry â€” existing item grows
- Visual feedback: brief overlay flash "Appended to last item"
- Appended content inherits existing tags
- CLI: `clipass copy --append / -a` or `echo "more" | clipass copy -a`

## Feature 6: Pause/Resume Monitoring

### Behavior

- `isPaused` state on clipboard monitor
- Menu bar: toggle "Pause Monitoring" / "Resume Monitoring"
- Menu bar icon appearance changes when paused (dimmed)
- CLI: `clipass pause` / `clipass resume`
- Pub/sub: emits `pause` and `resume` events
- Persists across restart (UserDefaults)
- `clipass status` shows paused/active state

## Feature 7: Hide Menu Bar Icon

- Pause/resume accessible via: `clipass pause/resume` CLI, or overlay UI indicator

## Technical Notes

### CLI Build Strategy

- `clipass-cli` is a separate `executableTarget` in Package.swift
- Shared protocol types (socket messages, argument structs) live in a `ClipassShared` library target used by both app and CLI
- App build phase copies the CLI binary into `clipass.app/Contents/MacOS/`

### SwiftData Threading

The socket server accepts connections on a background thread, but all SwiftData operations dispatch to `@MainActor`. Flow:
1. Socket receives command on background thread
2. Parses JSON into a typed `SocketCommand`
3. Dispatches to `@MainActor` closure that queries/mutates `modelContainer.mainContext`
4. Serializes response back on the socket thread

This uses the existing `mainContext` pattern â€” no separate ModelContext, no merge conflicts.

### Schema Migration

Adding `Tag` and `ActionChain` models requires:
- Adding both types to `ModelContainer(for:)` in AppServices
- SwiftData handles additive schema changes (new models, new optional fields) as lightweight migrations automatically
- The new `tags: [Tag]` relationship on ClipboardItem is additive (defaults to empty)

### Other

- Socket protocol uses 4-byte big-endian length-prefix framing (not newline-delimited)
- `clipass listen` converts framed messages to newline-delimited JSON on stdout
- `clipass listen` exits on broken pipe (no auto-reconnect) â€” scripts wrap in `while true` if needed
- Tags use SwiftData's implicit many-to-many (no explicit join table needed)
- Action chains store steps as JSON-encoded array in a single SwiftData field
- Copy-append hotkey uses the same KeyboardShortcuts framework as the overlay
