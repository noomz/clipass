# Phase 15 — UI Design Contract: Tags

> Visual and interaction contract for Phase 15 (Tags). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

<user_constraints>
## User Constraints (from CONTEXT.md)

### Locked Decisions
- Primary tag assignment interaction: context menu "Tag as..." submenu with checkmark toggles
- Available in BOTH overlay (right-click) and menu bar popup (HistoryItemRow context menu)
- Submenu lists all tags with checkmarks for assigned ones, plus "+ New Tag..." at bottom
- "+ New Tag..." triggers NSAlert with text field for name input, auto-assigns random color from palette
- No tag assignment in the inline editor — context menu only
- Tag badges on the SECOND line (metadata row) alongside source app and timestamp
- Style: colored dot + tag name in caption font (like GitHub labels)
- Max 3 visible tag badges per row, then "+N" overflow indicator
- Badges appear in BOTH overlay (OverlayItemRow) and menu bar popup (HistoryItemRow)
- `tag:name` prefix syntax in overlay search field (TAG-03)
- Multiple tag filters use OR logic: `tag:work tag:code` shows items with either tag
- Tag filters combine with free text: `tag:work hello` = items tagged 'work' AND containing 'hello'
- No autocomplete when typing `tag:` — user types full tag name manually
- New "Tags" tab in Settings (8th tab in sidebarAdaptable TabView)
- Layout: list + editor split pane (same pattern as Transforms/Actions tabs)
- Left side: tag list with colored dots, [+] [-] buttons
- Right side: name text field, preset color palette (8-10 colors), delete button
- Color picker: preset palette only (red, orange, yellow, green, blue, purple, pink, gray) — no system color picker
- No default starter tags — starts empty, users create what they need
- Deleting a tag requires confirmation alert: "Delete tag 'X'? This removes it from all items."

### Claude's Discretion
- Exact preset color hex values (should work well across all 5 themes)
- Tag model schema details (SwiftData implicit many-to-many already decided)
- Sort order of tags in context menu submenu and Settings list
- Exact badge spacing and layout within the metadata row
- Whether overflow "+N" is clickable or just informational

### Deferred Ideas (OUT OF SCOPE)
None — discussion stayed within phase scope
</user_constraints>

## Spacing Scale

Declared values follow the existing project's established spacing idioms (observed from `OverlayItemRow`, `HistoryItemRow`, `ContextActionsView`):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 2pt | Badge dot-to-name gap (HStack spacing: 3), badge vertical padding |
| sm | 4pt | Badge horizontal gap in metadata HStack, badge horizontal padding (5pt), row horizontal background padding |
| md | 6pt | Row vertical padding (theme.itemVerticalPadding), Settings action row vertical padding |
| lg | 12pt | Row horizontal padding (overlay), section padding |
| xl | 16pt | Settings content padding, AppearanceSettingsView VStack padding |

**Badge-specific spacing (Claude's discretion):**
- Dot-to-name gap: 3pt (`HStack(spacing: 3)`)
- Badge horizontal padding: 5pt (`.padding(.horizontal, 5)`)
- Badge vertical padding: 2pt (`.padding(.vertical, 2)`)
- Inter-badge gap: 4pt (`HStack(spacing: 4)` in metadata row)

**Settings Tags tab — left pane width:** 180pt (consistent with typical macOS Settings split-pane patterns)

Exceptions:
- Touch targets for [+] [-] buttons: minimum 24pt tap area (use `.buttonStyle(.plain)` with padding)
- Settings tab frame: 550 × 450pt (inherited from existing `SettingsView.frame`)

## Typography

All sizes use SF Pro (system font). Source: observed directly from `OverlayItemRow.swift`, `HistoryItemRow.swift`, `ContextActionsView.swift`, and `Theme.swift`.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (row preview text) | NSFont.systemFontSize (~13pt) | theme.titleFontWeight (.regular on most themes, .medium on Midnight) | 1.4 (system default) |
| Caption / metadata (source app, timestamp) | .caption2 (~10pt) | .regular (400) | 1.3 (system default) |
| Tag badge name | .caption2 (~10pt) | .regular (400) | 1.0 (single line, clipped) |
| Settings headline | .headline (~13pt semibold) | .semibold (600) | 1.4 |

**Tag badge font:** `.font(.caption2)` — matches the existing metadata row font used for source app and timestamp. Consistent with the "colored dot + tag name in caption font" decision.

**Overflow indicator font:** `.font(.caption2)` — same as badge name, using `theme.secondaryText` color (or `.white.opacity(0.75)` on selected rows).

## Color

The project uses a theme system (5 themes: System, Dark, Light, Midnight, Nord). Tag badge colors must work across all 5. There is no single hex for "dominant/secondary/accent" — instead, colors reference theme tokens.

| Role | Token / Value | Usage |
|------|--------------|-------|
| Dominant surface | `theme.overlayBackground` / `Color.clear` (vibrancy) | Overlay panel background — unchanged |
| Secondary surface | `theme.searchFieldBackground` | Search field, selected row highlight background |
| Selection accent | `theme.itemBackground` (per-theme accent) | Selected row highlight — unchanged |
| Primary text | `theme.primaryText` | Row content preview text |
| Secondary text | `theme.secondaryText` | Source app, timestamp, overflow "+N" indicator |
| Destructive | `.red` / SwiftUI `.destructive` role | Delete button in Settings Tags editor, tag delete confirmation |
| Pin indicator | `.orange` (hardcoded — existing convention) | Pin badge — unchanged |

**Tag badge colors (preset palette — Claude's discretion, must work on all 5 themes):**

| Slot | Hex | Name |
|------|-----|------|
| 1 | `#e05252` | Red |
| 2 | `#e0873a` | Orange |
| 3 | `#d4c140` | Yellow |
| 4 | `#4db86a` | Green |
| 5 | `#3a8fd4` | Blue |
| 6 | `#6b52d4` | Purple |
| 7 | `#cc52a8` | Pink |
| 8 | `#888888` | Gray |

**Color application rules for tag badges:**
- Badge dot: `Color(hex: tag.colorHex)` — full opacity circle, 6×6pt
- Badge text: `Color(hex: tag.colorHex)` on unselected rows; `.white.opacity(0.85)` on selected rows
- Badge capsule background: `Color(hex: tag.colorHex).opacity(0.15)` on unselected rows; `Color(hex: tag.colorHex).opacity(0.30)` on selected rows

**Rationale for selection inversion:** On selected rows, the row background becomes `theme.itemBackground` (a solid accent color). The badge capsule at 0.15 opacity would be nearly invisible against that background. Bumping to 0.30 maintains visual contrast while keeping the badge readable on any theme's selection color.

Accent reserved for: overlay row selection highlight, pencil icon in overlay rows, Settings `Color.accentColor` hover states — NOT for tag badge colors (tag badges use their own per-tag colors).

## Component Inventory

Components introduced or modified in this phase:

### New: TagBadgeView

A reusable badge rendering a colored dot and tag name inside a capsule.

```
[●  work]   ← dot (6×6pt) + name (caption2) inside Capsule background
```

**Anatomy:**
- Outer shape: `Capsule` fill at `tagColor.opacity(0.15)` (or 0.30 on selected rows)
- Dot: `Circle().fill(tagColor)` — 6×6pt
- Label: `Text(tag.name)` at `.caption2`, `tagColor` color (`.white.opacity(0.85)` on selected)
- HStack spacing: 3pt
- Padding: `.horizontal 5pt`, `.vertical 2pt`

**Props:** `tag: Tag`, `isSelected: Bool`

### New: TagBadgesRow

A horizontal strip of up to 3 `TagBadgeView` instances plus an optional "+N" overflow label, sorted alphabetically by tag name to ensure deterministic render order.

**Anatomy:**
- `HStack(spacing: 4)` containing `ForEach(item.tags.sorted { $0.name < $1.name }.prefix(3))`
- Followed by `Text("+\(overflowCount)")` at `.caption2` in `theme.secondaryText` if overflow > 0
- Overflow indicator: informational only — not tappable (out of scope for this phase)

**Insertion point in OverlayItemRow:** Between source app and timestamp in the existing metadata `HStack`. The metadata HStack becomes a two-sub-HStack layout:

```
[sourceApp]  [tag1][tag2][+1]  [Spacer]  [timestamp]
```

**Insertion point in HistoryItemRow:** Same metadata HStack pattern. Between sourceApp and timestamp.

### New: TagListRow (Settings left pane row)

A row in the Settings Tags list showing a colored dot and tag name.

**Anatomy:**
- `HStack(spacing: 8)`: `Circle().fill(Color(hex: tag.colorHex))` (10×10pt) + `Text(tag.name)` at `.body`
- Uses `List` selection binding (`.tag(tag)` modifier)

### New: TagEditorPane (Settings right pane)

Inline editor for a selected tag. Uses `@Bindable` for mutation without a sheet.

**Anatomy:**
- `Form { TextField("Name", text: $tag.name) }` with `.textFieldStyle(.roundedBorder)`
- `TagColorPalette` component (see below)
- `Button("Delete Tag", role: .destructive)` — triggers confirmation alert

### New: TagColorPalette

An 8-swatch horizontal color picker using the preset palette only.

**Anatomy:**
- `HStack(spacing: 8)` of 8 `Circle` buttons
- Each circle: 22×22pt, filled with preset color
- Selected swatch: checkmark overlay (`Image(systemName: "checkmark")` in white, `.caption2`)
- Not-selected swatch: no border (color is self-explanatory)

### New: TagsView (Settings tab)

The full Settings > Tags tab using an `HStack` split-pane layout.

**Anatomy:**
- Left pane (180pt wide): `List(selection: $selectedTag)` + bottom toolbar with `+` and `-` buttons
- Divider (1pt vertical)
- Right pane (fills remaining width): `TagEditorPane` or "Select a tag" empty state
- Bottom toolbar uses `.buttonStyle(.plain)` + `padding(6)` — matching existing `RulesView` toolbar conventions

### Modified: OverlayItemRow

- Second metadata line: insert `TagBadgesRow` between sourceApp and timestamp
- Context menu: add `Menu("Tag as...")` submenu with per-tag `Button` checkmarks + `Divider` + `Button("+ New Tag...")`
- `Menu("Tag as...")` placement: after Pin/Unpin, before Copy As...

### Modified: HistoryItemRow

- Second metadata line: insert `TagBadgesRow` between sourceApp and timestamp
- Context menu: add flat tag buttons (NOT inside a `Menu`) due to MenuBarExtra submenu bug
  - Each tag renders as `Button(item.tags.contains(tag) ? "✓ \(tag.name)" : "   \(tag.name)")` directly in `.contextMenu`
  - `Button("+ New Tag...")` follows the tag buttons
  - Section placement: after Pin/Unpin, before Copy As...
  - Flat buttons grouped with a `Divider` before and after the tag block

## Interaction Contract

### Tag assignment (overlay — NSPanel)

1. User right-clicks a row in overlay
2. Context menu appears with `Menu("Tag as...")` entry
3. User hovers `Menu("Tag as...")` to expand submenu
4. Submenu shows: each tag with checkmark (if assigned) or blank, then `---`, then `+ New Tag...`
5. User clicks a tag name to toggle assignment (add if unassigned, remove if assigned)
6. Badge row on the row updates immediately (SwiftData `@Query` live binding)
7. User clicks `+ New Tag...` → `NSAlert` with text field appears
8. User types name, clicks OK → new tag created with random color from palette, auto-assigned to item

### Tag assignment (menu bar popup — MenuBarExtra)

Same flow, but `Menu("Tag as...")` submenu is replaced with flat `Button` items directly in `.contextMenu`:
- Each tag shown as `Button("✓ work")` or `Button("   work")` (text-prefix checkmarks)
- `Button("+ New Tag...")` at end of tag block
- Due to MenuBarExtra submenu bug: NO `Menu` wrapper

### Tag filtering (overlay search)

1. User types `tag:work` in overlay search field
2. `filteredItems` parses `tag:` tokens and applies OR matching
3. Items without the `work` tag are filtered out
4. User can combine: `tag:work hello` → must have tag "work" AND contain "hello"
5. Multiple tags: `tag:work tag:code` → has "work" OR "code"
6. No autocomplete — user types full tag name manually

### Tag management (Settings > Tags)

1. User opens Settings, selects "Tags" tab (8th)
2. Left pane shows alphabetically sorted tag list (empty state if no tags)
3. User clicks `+` → new tag created (name: "New Tag", color: first palette color), auto-selected for editing
4. User edits name in right pane text field — change persists on each keystroke via `@Bindable`
5. User selects color swatch — badge preview in left list updates immediately
6. User clicks `−` or "Delete Tag" → confirmation alert fires
7. On confirm: tag deleted from all items (SwiftData `.nullify` cascade), removed from list

### Confirmation dialog for tag deletion

Uses SwiftUI `.confirmationDialog` (consistent with existing `RulesView.confirmationDialog` pattern):
- Title: "Delete tag '\(tag.name)'?"
- Message: "This removes it from all items."
- Destructive button: "Delete"
- Cancel button: "Cancel"
