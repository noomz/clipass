# Architecture Research: v1.1 More Control

**Project:** clipass
**Researched:** 2026-02-06
**Scope:** Integration patterns for v1.1 settings features
**Confidence:** HIGH (based on direct codebase analysis)

## Executive Summary

v1.1 features integrate cleanly with the existing architecture. The codebase already has:
- A Services layer pattern (ClipboardMonitor, TransformEngine, HookEngine)
- SwiftData models with consistent patterns
- A tabbed Settings window ready for extension
- Clear separation between monitoring, processing, and display

The integration strategy: add new Models for settings, extend ClipboardMonitor for filtering, create a RedactionEngine service, and add Settings tabs.

## Existing Architecture Overview

```
clipassApp.swift
    └── AppServices (singleton)
            ├── modelContainer (SwiftData: ClipboardItem, TransformRule, Hook)
            ├── clipboardMonitor ─┬── polls pasteboard every 500ms
            │                     ├── filters by hardcoded ignoredTypes
            │                     ├── calls transformEngine.transform()
            │                     ├── saves to SwiftData
            │                     └── calls hookEngine.executeHooks()
            ├── transformEngine ──── applies regex rules from TransformRule
            └── hookEngine ───────── runs shell commands from Hook

Views/
    ├── ClipboardPopup.swift ── @Query ClipboardItem, displays HistoryItemRow
    ├── HistoryItemRow.swift ── shows item.content.prefix(50)
    ├── SettingsView.swift ──── TabView with RulesView, HooksView
    ├── RulesView.swift
    ├── RuleEditorView.swift
    ├── HooksView.swift
    └── HookEditorView.swift

Models/
    ├── ClipboardItem.swift ─── id, content, sourceApp, timestamp
    ├── TransformRule.swift ─── id, name, pattern, replacement, sourceAppFilter, isEnabled, order
    └── Hook.swift ───────────── id, name, pattern, command, sourceAppFilter, isEnabled, order
```

## Integration Points

### 1. Ignored Apps (Replace Hardcoded List)

**Current state (ClipboardMonitor.swift:16-20):**
```swift
private let ignoredTypes = [
    "org.nspasteboard.TransientType",
    "org.nspasteboard.ConcealedType",
    "org.nspasteboard.AutoGeneratedType"
]
```

**Integration point:** `ClipboardMonitor.poll()` at line 53-64

**Strategy:** 
- Keep `ignoredTypes` as immutable (security, not configurable)
- Add new check for `IgnoredApp` model matching `sourceApp` bundle ID
- Query `IgnoredApp` from SwiftData in poll loop

**Data flow change:**
```
poll() → check pasteboard types (security) 
       → check ignored apps (NEW: user-configurable)
       → check ignored patterns (NEW)
       → continue with content processing
```

### 2. Content Patterns to Skip (New Filtering)

**Integration point:** `ClipboardMonitor.poll()` after getting content (line 66)

**Strategy:**
- After `pasteboard.string(forType: .string)` returns content
- Query `IgnoredPattern` models
- Test content against each pattern regex
- Skip storing if any pattern matches

**Note:** This is BEFORE transforms apply (we don't want to transform-then-discard).

### 3. Display Formatting (Invisible Chars, Truncation)

**Current state (HistoryItemRow.swift:12):**
```swift
Text(item.content.prefix(50) + (item.content.count > 50 ? "..." : ""))
```

**Integration point:** `HistoryItemRow` view rendering

**Strategy:**
- Create `DisplaySettings` model (singleton pattern in SwiftData)
- Create `RedactionEngine` service for display processing
- `HistoryItemRow` queries `DisplaySettings` and applies formatting
- Invisible char cleaning applied before truncation

**Display processing pipeline:**
```
item.content 
  → RedactionEngine.cleanForDisplay(content, settings) 
  → apply truncation per settings.previewLength
  → render in Text view
```

### 4. Sensitive Content Redaction

**Integration point:** Same as display formatting (HistoryItemRow)

**Strategy:**
- `RedactionEngine` handles both invisible chars AND sensitive pattern redaction
- Sensitive patterns stored in `DisplaySettings.sensitivePatterns` or separate model
- Redaction only affects DISPLAY, not stored content
- User can still copy full content (redaction is preview-only)

**Redaction examples:**
- Email: `user@example.com` → `u***@e***.com`
- Possible password: `password123` → `p*******3`

### 5. History Limits (Max Items, Auto-Cleanup)

**Current state (ClipboardMonitor.swift:14, 117-130):**
```swift
let maxItems: Int = 100
// ... pruneOldItems uses this hardcoded value
```

**Integration point:** 
- `ClipboardMonitor.pruneOldItems()` for max items
- New scheduled cleanup for age-based pruning

**Strategy:**
- `DisplaySettings` or `HistorySettings` model stores limits
- `ClipboardMonitor` reads settings instead of hardcoded value
- Age-based cleanup runs on app start and periodically

### 6. Hotkey Customization

**Current state (clipassApp.swift:5-7):**
```swift
extension KeyboardShortcuts.Name {
    static let toggleClipboard = Self("toggleClipboard", default: .init(.v, modifiers: [.command, .shift]))
}
```

**Integration point:** Already configurable via KeyboardShortcuts library

**Strategy:**
- KeyboardShortcuts library already persists user customization
- Add UI in Settings to change hotkey using `KeyboardShortcuts.Recorder`
- No new model needed (library handles persistence)

### 7. Start on Login

**Integration point:** New functionality, macOS Login Items API

**Strategy:**
- Use `SMAppService` (macOS 13+) for login item registration
- Store preference in `AppSettings` model
- Toggle in Settings UI

## New Components

### Models (4 new files)

| Model | Purpose | Fields |
|-------|---------|--------|
| `IgnoredApp.swift` | Apps to skip capturing from | id, bundleIdentifier, name (display), isEnabled |
| `IgnoredPattern.swift` | Content patterns to skip storing | id, name, pattern (regex), isEnabled |
| `DisplaySettings.swift` | Display/preview settings (singleton) | cleanInvisibleChars, previewLength, sensitivePatterns (JSON array) |
| `AppSettings.swift` | App behavior settings (singleton) | maxHistoryItems, maxHistoryAgeDays, startOnLogin |

**Why separate models vs one giant Settings model:**
- `IgnoredApp` and `IgnoredPattern` are lists (0-N items)
- `DisplaySettings` and `AppSettings` are singletons (exactly 1 each)
- Separation allows different UI patterns (list vs form)

### Services (1 new file)

| Service | Purpose |
|---------|---------|
| `RedactionEngine.swift` | Clean invisible chars, apply sensitive pattern redaction for display |

**Why a service:**
- Testable in isolation
- Reusable if other views need display processing
- Follows existing pattern (TransformEngine, HookEngine)

```

    → TransformEngine.transform()
    → Save ClipboardItem
    → HookEngine.executeHooks()

    → @Query ClipboardItem
    → HistoryItemRow shows prefix(50)
```

### After (v1.1)

```

    → ClipboardMonitor.poll()
    → Check hardcoded ignoredTypes (unchanged)
    → Check IgnoredApp model (NEW)
    → Check IgnoredPattern model (NEW)
    → TransformEngine.transform() (unchanged)
    → Save ClipboardItem
    → Prune with configurable limit (MODIFIED)
    → HookEngine.executeHooks() (unchanged)

    → @Query ClipboardItem
    → @Query DisplaySettings
    → RedactionEngine.cleanForDisplay() (NEW)
    → HistoryItemRow shows processed content with configurable length
```

## Suggested Build Order

Based on dependencies and testability:

### Phase 1: Foundation Models & Settings UI Shell
**Goal:** Data layer for all settings, basic UI structure

1. Create `IgnoredApp.swift` model
2. Create `IgnoredPattern.swift` model  
3. Create `DisplaySettings.swift` model (singleton pattern)
4. Create `AppSettings.swift` model (singleton pattern)
5. Update `clipassApp.swift` to register new models
6. Add placeholder tabs to `SettingsView.swift`

**Why first:** All other features depend on these models existing.

### Phase 2: Ignored Apps Feature
**Goal:** User can manage ignored apps, filtering works

1. Create `IgnoredAppsView.swift` with list UI
2. Create `IgnoredAppEditorView.swift` for add/edit
3. Modify `ClipboardMonitor.swift` to check IgnoredApp
4. Test filtering behavior

**Why second:** Simplest filtering feature, validates the pattern.

### Phase 3: Ignored Patterns Feature
**Goal:** User can manage content patterns to skip

1. Create `IgnoredPatternsView.swift` with list UI
2. Create `IgnoredPatternEditorView.swift` for add/edit
3. Modify `ClipboardMonitor.swift` to check IgnoredPattern
4. Test content pattern filtering

**Why third:** Similar pattern to Phase 2, builds on same filtering code.

### Phase 4: Display Settings & Redaction
**Goal:** Configurable display with redaction

1. Create `RedactionEngine.swift` service
2. Create `DisplaySettingsView.swift` UI
3. Modify `HistoryItemRow.swift` to use RedactionEngine
4. Implement invisible char cleaning
5. Implement sensitive pattern redaction
6. Test display processing

**Why fourth:** Display-only changes, no impact on data storage.

### Phase 5: History & App Settings
**Goal:** Configurable history limits, start on login, hotkey UI

1. Create `GeneralSettingsView.swift` with all general options
2. Modify `ClipboardMonitor.pruneOldItems()` to use settings
3. Add age-based cleanup logic
4. Add KeyboardShortcuts.Recorder for hotkey
5. Add SMAppService for login item
6. Test history limits and app behavior

**Why last:** Independent features that don't block others.

## Singleton Pattern for Settings

For `DisplaySettings` and `AppSettings`, use SwiftData singleton pattern:

```swift
@Model
class DisplaySettings {
    static let defaultId = UUID(uuidString: "00000000-0000-0000-0000-000000000001")!
    
    var id: UUID = DisplaySettings.defaultId
    var cleanInvisibleChars: Bool = true
    var previewLength: Int = 50
    var sensitivePatterns: String = "[]"  // JSON array
    
    static func shared(context: ModelContext) -> DisplaySettings {
        let descriptor = FetchDescriptor<DisplaySettings>(
            predicate: #Predicate { $0.id == DisplaySettings.defaultId }
        )
        if let existing = try? context.fetch(descriptor).first {
            return existing
        }
        let settings = DisplaySettings()
        context.insert(settings)
        return settings
    }
}
```

## Risk Considerations

| Risk | Mitigation |
|------|------------|
| Performance: querying settings on every poll | Cache settings in ClipboardMonitor, refresh on change notification |
| SwiftData: 4 new models in container | Test migration from v1.0 database |
| Complexity: too many Settings tabs | Consider grouping (General tab covers multiple) |
| Redaction: false positives | Make patterns user-configurable, provide good defaults |
